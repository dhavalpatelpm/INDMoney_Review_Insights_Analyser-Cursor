# INDmoney Review Insights Analyzer — End-to-End Architecture

## 1. Problem Statement

**Product:** [INDmoney](https://play.google.com/store/apps/details?id=in.indwealth&hl=en_IN) — NRI Investment App (Indian & US Stocks, SIP, Flash Trading)

**Goal:** Turn recent App Store / Play Store reviews into a **one-page weekly pulse** containing:
- **Top themes** (what users talk about most)
- **Real user quotes** (representative, anonymised)
- **Three action ideas** (prioritised next steps)
- **Draft email** with this weekly note (recipient email from front-end or config)

**Value:**
- **Product / Growth Teams** → understand what to fix next  
- **Support Teams** → know what users are saying & acknowledging  
- **Leadership** → quick weekly health pulse  

---

## 2. Scope & Constraints

| Requirement | Detail |
|-------------|--------|
| **Review window** | Last 8–12 weeks |
| **Review fields** | Rating, title, text, date |
| **Themes** | 3–5 themes max; generated by LLM, then used to group reviews |
| **Weekly note** | Top 3 themes, 3 user quotes, 3 action ideas |
| **Output** | One-page note + draft email (recipient from front-end or config/env) |
| **PII** | Do **not** include; anonymise/redact in outputs |
| **Review source** | Google Play (primary); design allows future App Store) |
| **Scraper** | `google-play-scraper` library |
| **LLM** | Groq (Phase 2); Gemini (Phase 3) |

## 4. Phase-Wise Architecture

### Phase 1 — Review Ingestion and Cleaning

**Objective:** Pull Play Store reviews for the last 8–12 weeks, persist them, then clean and redact PII so downstream phases only see safe text.

| Component | Responsibility |
|-----------|----------------|
| **Review Fetcher** | Uses `google-play-scraper` to fetch reviews for `in.indwealth`. Paginate until date range (e.g. 12 weeks back) is covered. |
| **Pagination** | Handle `continuationToken` (or equivalent) to get enough reviews; stop when oldest review is older than 12 weeks. |
| **Data model** | Store: `review_id`, `rating`, `title`, `text`, `date`, `thumbs_up` (optional), `app_version` (if available). |
| **Persistence** | Local/cloud storage: e.g. JSON/CSV file, or SQLite/DB for deduplication and re-runs. Prefer idempotent writes (upsert by `review_id`). |
| **Text cleaning** | Normalise encoding, trim, collapse whitespace; optional: remove app-specific boilerplate. |
| **PII redaction** | Redact/replace emails, phone numbers, names, etc. in `text`. Store cleaned+redacted version for Phase 2+. |
| **Language filter** | Keep only English reviews (e.g. langdetect); drop non-English before saving cleaned. |
| **Config** | `APP_ID = "in.indwealth"`, `WEEKS_BACK = 12` (or 8), output path for raw and cleaned reviews. |

**Data flow:**
```
Google Play Store (via google-play-scraper)
        → Raw reviews (rating, title, text, date, …)
        → Dedupe by review_id → Persist raw
        → Clean text → PII redact
        → Persist cleaned reviews (file / DB)
```

**Considerations:**
- Rate limiting / backoff to avoid blocking.
- Optional: run on schedule (e.g. weekly) and append new reviews.

---

### Phase 2 — Theme Discovery and Classification (Groq)

**Objective:** Use Groq to discover 3–5 themes from cleaned reviews, assign each review to a theme, and group/rank them.

| Component | Responsibility |
|-----------|----------------|
| **Load reviews** | Read cleaned (and PII-redacted) reviews from Phase 1; filter by date if needed. |
| **Theme discovery (Groq)** | **Groq**: One shot over a sample or summarised set of reviews. Prompt: “Given these review summaries/texts, suggest 3–5 distinct thematic labels (e.g. ‘Chart/UX’, ‘Fractional shares’, ‘Sync/Performance’). Return only theme names.” |
| **Theme classification (Groq)** | For each review (or batched): Groq assigns one theme from the list. Input: theme list + review text; output: single theme label. |
| **Grouping** | Group reviews by assigned theme; rank themes by frequency or impact. |
| **Output** | Structured data: `theme → [review_ids]`, and for each review: cleaned text, rating, date. |

**Data flow:**
```
Cleaned reviews (from Phase 1)
    → Groq: discover 3–5 themes (once)
    → Groq: assign theme per review (batch)
    → Group by theme
    → Rank themes (e.g. by count or impact)
```

**Considerations:**
- Groq API key in env/config; no key in code.
- Batch size for theme assignment to stay within context limits and cost.
- Fallback theme “Other” for low-confidence or irrelevant reviews.

---

### Phase 3 — Weekly Note Generation (Gemini)

**Objective:** Produce a single “weekly pulse” document: top 3 themes, 3 user quotes, 3 action ideas (using **Gemini** for action ideas).

| Component | Responsibility |
|-----------|----------------|
| **Theme selection** | Pick top 3 themes from Phase 2 theme_rank (by count). |
| **Quote selection** | From top 3 themes: choose 1 representative quote per theme (or 3 total). Heuristic: clear, mid-length, impactful; no PII. |
| **Action ideas** | **Gemini**: Input = top themes + representative quotes. Prompt: “Suggest 3 concrete, prioritised action ideas for Product/Support/Leadership.” Output = 3 short action items. |
| **One-page note** | Template: Title (e.g. “INDmoney Weekly Review Pulse — &lt;date_range&gt;”), Top 3 themes (with 1 quote each), 3 action ideas, review count, date range. All text PII-free. |
| **Structured output** | JSON/dict: `themes`, `quotes`, `actions`, `meta` (date_range, total_reviews). Saved as JSON + human-readable .txt. |

**Data flow:**
```
Theme groups (from Phase 2)
    → Select top 3 themes
    → Select 3 quotes (1 per theme or best 3)
    → Gemini: 3 action ideas
    → Fill one-page note template
    → Structured note (JSON + .txt) in data/weekly_notes/
```

---

### Phase 4 — Email Delivery

**Objective:** Turn the weekly note into a **personalized** draft email and send it to the recipient. **Recipient email and name are supplied by the front-end** (or, for CLI, from config/env).

| Component | Responsibility |
|-----------|----------------|
| **Email draft** | Body = personalized greeting **“Hi, {recipient_name},”** (when name provided) then one-page note (plain text). Subject = e.g. “INDmoney Weekly Pulse — &lt;date_range&gt;”. No PII. |
| **Recipient** | **From front-end:** pass `recipient_email` and `recipient_name` to `run_phase4(recipient_email=..., recipient_name=...)`. For CLI: optional `email_to` and `recipient_name` in config or `EMAIL_TO` in env. |
| **Send** | Use SMTP (Gmail/Outlook, etc.) with credentials from env. Recipient = `recipient_email` (front-end) or config/env fallback. |
| **Optional** | “Dry run” mode: only write draft to file (no send). |

**Data flow:**
```
Weekly note (text + meta)
    → Email body + subject
    → Recipient from front-end (run_phase4(recipient_email=...)) or config/env
    → SMTP / Email API → Sent to recipient
```

**Considerations:**
- SMTP credentials (SMTP_USER, SMTP_PASSWORD, etc.) in environment variables or secret manager.
- Recipient: email and name provided by the front-end when calling Phase 4 (`recipient_email`, `recipient_name`); CLI can use `EMAIL_TO` and optional `recipient_name` in config.
- Optional: “dry run” mode that only writes the draft to a file (no send).

## 8. LLM Usage Summary

**Groq (Phase 2):** theme generation, theme assignment per review.

**Gemini (Phase 3):** action ideas from top themes + quotes.

| Step | LLM | Purpose | Input | Output |
|------|-----|--------|--------|--------|
| 1 | Groq | Theme generation | Sample/summary of reviews (redacted) | 3–5 theme labels |
| 2 | Groq | Theme assignment | Theme list + each review text (redacted) | One theme per review |
| 3 | — | Quote selection | Top themes + groups | 3 quotes (heuristic) |
| 4 | **Gemini** | Action ideas | Top themes + quotes + context | 3 action items |

All prompts must enforce: no PII in output; themes and actions in English; short, product-actionable format.

## 9. Security & Privacy

- **PII:** Redact before any Groq call and before writing to weekly note or email. Do not log raw review text with PII.
- **Secrets:** Groq API key (Phase 2), Google/Gemini API key (Phase 3), SMTP credentials in `.env` or secret manager; never in repo. **Recipient email** is supplied by the front-end at send time (not stored as secret).
- **Outputs:** Weekly note and email must be safe to share internally (no user identifiers, emails, or phone numbers).

## 10. Web UI (Trigger & View)

A browser-based interface allows users to run the pipeline and control email delivery without using the CLI.

| Component | Responsibility |
|-----------|----------------|
| **App entry** | Streamlit app (e.g. `streamlit run app/streamlit_app.py`) or FastAPI + minimal frontend. |
| **Generate one-pager** | Button/action that runs Phase 1 → 3 (or 2 → 3 if cleaned data exists). Displays the weekly note in the UI: top 3 themes, 3 quotes, 3 action ideas. Optionally saves to `data/weekly_notes/`. |
| **View one-pager** | Show the latest generated note in the session, or load from last saved file. Read-only display; no PII. |
| **Send email** | Button that runs Phase 4 (uses current/latest note). **User enters recipient email and recipient name** in the UI; front-end calls `run_phase4(recipient_email=..., recipient_name=...)` for a personalized “Hi, {name},” greeting. Optional “Dry run” to preview. |
| **Optional controls** | Sidebar or form: weeks back (8–12), “Use cached reviews” to skip Phase 1, **recipient email** and **recipient name** inputs (name used for greeting). |

**Flow:**
1. User opens UI in browser.
2. Clicks “Generate one-pager” → pipeline runs (Phases 1–3); one-page note appears on screen.
3. User enters recipient email and recipient name, then clicks “Send email” → Phase 4 runs with `recipient_email=...`, `recipient_name=...` (email body starts with “Hi, {name},”); confirmation or error shown.
4. Alternatively: “Send email” only (if note was already generated or loaded).

**Tech:** Streamlit recommended for speed (single Python app, no separate frontend). Pipeline code is reused; UI only invokes `run_pipeline` or equivalent step functions.

## 11. Execution Model & Triggers

The pipeline (Phase 1 → 4) can be triggered in three ways:

| Trigger | Use case |
|--------|----------|
| **CLI** | `python run_pipeline.py` (or similar) runs full pipeline; suitable for cron or one-off scripts. |
| **Web UI** | Browser-based interface to run the same pipeline and control output. |
| **Scheduler** | `python run_scheduler.py` — runs pipeline **every 5 minutes** (15:40, 15:45, 15:50, … IST); emails to fixed recipient (`dhavalpatel.sdds@gmail.com`). Uses `run_pipeline.py` CLI. Requires `APScheduler`. Logs to `logs/scheduler.log`. Runs until stopped (Ctrl+C). |

**Web UI behaviour:**
- **Generate one-pager:** Trigger Phase 1 → 3 (ingest → themes → weekly note). Show the one-page note in the UI (themes, quotes, action ideas). Optionally save to `data/weekly_notes/`.
- **Send email:** After generating (or loading latest note), user enters recipient email and name in the UI; trigger Phase 4 with `run_phase4(recipient_email=..., recipient_name=...)` to send the personalized draft. Optional “dry run” to preview only.
- **Optional UI controls:** Configure weeks back (8–12), view last run meta (date range, review count), re-run from cached cleaned reviews to skip Phase 1.

**Implementation notes:**
- `run_pipeline.py` remains the single orchestration entry point; the UI and scheduler invoke it (or the same step functions) so logic is not duplicated.
- **One-shot (CLI):** `run_pipeline.py` runs Phase 1 → 4 in sequence.
- **Scheduler:** `run_scheduler.py` uses APScheduler (timezone: Asia/Kolkata). Runs every 5 min; invokes `run_pipeline.py --phase 1,2,3,4 --email dhavalpatel.sdds@gmail.com --name "Dhaval Patel"`. Logs to `logs/scheduler.log`. Requires `email_dry_run: false` and SMTP credentials in `.env`.
- **Dry run:** Config flag (or UI toggle) to skip email send and only write draft to `data/weekly_notes/`.

## 12. Success Criteria

- Reviews from last 8–12 weeks ingested without PII in logs.
- 3–5 themes generated by Groq; reviews grouped; top 3 themes in the note.
- Exactly 3 user quotes and 3 action ideas on the one-page note.
- One draft email sent to the recipient (email provided by user in front-end, or EMAIL_TO/config for CLI), with no PII.
- **UI:** User can trigger “Generate one-pager” and see the note in the browser; user enters recipient email and name and triggers “Send email” to deliver the personalized note (“Hi, {name},” + weekly pulse).
